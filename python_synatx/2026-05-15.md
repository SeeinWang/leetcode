## `collections.deque`

`deque` = **double-ended queue**（双端队列），来自 `collections` 模块。两端进出都是 **O(1)**，而普通 `list` 在头部操作是 O(n)。

### 核心操作

```python
from collections import deque

d = deque([1, 2, 3])

# 右端（尾部）
d.append(4)        # [1,2,3,4]      O(1)
d.pop()            # 4, 剩 [1,2,3]  O(1)

# 左端（头部）
d.appendleft(0)    # [0,1,2,3]      O(1)
d.popleft()        # 0, 剩 [1,2,3]  O(1)

# 其它
len(d)             # 大小 O(1)
d[0]               # 两端索引 O(1)
d.extend([5,6])    # 右侧批量加
d.extendleft([0])  # 左侧加（注意顺序会反）
```

### BFS 为什么必须 deque

```python
# list
queue = [root]
queue.pop(0)       # ❌ O(n)，整体前移

# deque
queue = deque([root])
queue.popleft()    # ✅ O(1)
```

队列本质是 FIFO：一端进、另一端出。`list` 头部出队让整个 BFS 退化到 O(n²)。

### 底层原理

`deque` 是用**双向链表 + 分块数组**实现的（CPython 里是 block 链表，每块 64 个元素）。两端是指针操作 → O(1)；中间索引慢 → O(n)。

### 随机访问能力

| 操作 | `list` | `deque` |
|---|---|---|
| `d[0]` / `d[-1]` | O(1) | **O(1)**（两端特殊优化） |
| `d[中间]` | O(1) | **O(n)** |
| `d[i] = x` 中间赋值 | O(1) | O(n) |
| 切片 `d[1:4]` | O(k) | ❌ **不支持**，TypeError |
| `d.sort()` | ✅ | ❌ 没有 |
| `d.index(x)` | O(n) | O(n) |

要切片或排序得先转回 list：`list(d)[1:3]`。

### 常用初始化

```python
deque()                  # 空
deque([1,2,3])           # 从可迭代对象
deque(maxlen=5)          # 定长，满了从另一端挤掉（滑动窗口、最近 N 个）
```

### LeetCode 典型用途

1. **BFS / 层序遍历**（102, 199, 994 腐烂橘子）
2. **滑动窗口最大值**（239，单调 deque）
3. **回文判断**：两端同时 `popleft` / `pop` 对比
4. **最近 K 个元素**：`maxlen`

### Stack 要不要用 deque？

**不需要，直接用 `list`**。栈只在一端进出，`list` 尾部就是 O(1)：

```python
stack = []
stack.append(x)   # push, O(1)
stack.pop()       # pop,  O(1)
stack[-1]         # peek, O(1)
```

deque 也能当栈，但没有任何优势，反而：
- 多一行 import
- 常数稍慢（block 链表 vs 连续数组）
- 不支持切片，调试 / 打印不如 list 方便

### 一句话总结

> **栈 → list**；**队列 / 双端 → deque**。需要"两端都快"才用 deque。

## `for x in EXPR` 的迭代器机制 & "快照"问题

### `for` 的真实展开

```python
for x in EXPR:
    body
```

等价于：

```python
_it = iter(EXPR)        # ← EXPR 只在这里求值一次
while True:
    try:
        x = next(_it)
    except StopIteration:
        break
    body
```

`EXPR` 只被求值一次，循环体里再怎么改外部状态，**已经建立的迭代器**自己怎么吐元素由它决定。

### 关键：是不是"快照"取决于 EXPR 本身

不是 `for` 提供快照，而是**迭代对象自己**决定。

#### ① 静态快照型 ✅

```python
q = [1, 2, 3]
r = range(len(q))   # range(0, 3)，参数已固化
q.append(99)
for i in r:         # 还是 0,1,2，不受影响
    print(i)
```

属于这类的：
- `range(...)` — 创建时存死 start/stop/step
- `tuple(...)` — 不可变
- `list(q)` — 复制出来的新 list
- 已经物化的序列

#### ② 动态视图型 ⚠️

**直接迭代 list / deque / dict / set 本身，迭代器绑定到原对象**：

```python
q = [1, 2, 3]
for x in q:
    q.append(99)        # ❌ list 不报错但行为未定义（可能死循环）

from collections import deque
dq = deque([1, 2, 3])
for x in dq:
    dq.append(99)       # RuntimeError: deque mutated during iteration

d = {'a': 1}
for k in d:
    d['b'] = 2          # RuntimeError: dictionary changed size during iteration
```

#### ③ 惰性求值型 ⚠️

生成器 / `map` / `filter` / `zip` — 每次 `next()` 才算下一个，会看到上游变化：

```python
q = [1, 2, 3]
g = (x * 2 for x in q)   # 还没开始算
q.append(10)
list(g)                  # [2, 4, 6, 20] ← 看到了新加的
```

### `while` 没有快照保护

```python
while len(queue) > 0:    # 每轮重新求值
    ...
```

`while` 的条件每轮都重算，所以 `while len(queue) > 0` = "队列非空就一直跑"，分不出层。

### BFS 写法对比

```python
# ✅ 快照：range 是静态的，BFS 惯用法
for _ in range(len(queue)):
    node = queue.popleft()
    queue.append(...)    # 安全，不影响本轮

# ❌ 直接迭代 deque：边迭代边改
for node in queue:
    queue.append(...)    # RuntimeError

# ✅ 也行：先复制一份（浪费 O(n) 空间）
for node in list(queue):
    queue.append(...)
queue.clear()

# ✅ 也行：用计数变量（等价于 range 写法）
size = len(queue)
for _ in range(size):
    ...
```

### 一句话

> `for ... in X` 的"快照性"不是 `for` 给的，是 **X 自己**给的。`range` / `tuple` / `list(q)` 复制是快照；**直接迭代原 list / deque / dict 不是**。BFS 写 `range(len(queue))` 既快又安全。
